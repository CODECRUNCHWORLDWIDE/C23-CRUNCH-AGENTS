# Lecture 2 — Document Extraction and the A/B Methodology

> **Duration:** ~2 hours of reading + hands-on.
> **Outcome:** You can choose the right extraction tool for a document by its type (PyMuPDF / Unstructured / MinerU / LlamaParse), run OCR with Tesseract and Surya on scanned pages, handle tables and multimodal pages without silently dropping content, and run the chunking A/B harness that fixes the embedding and store, varies only the chunker, and reports MRR / Recall@5 / faithfulness to pick a winner with reasons.

Lecture 1 was *how to cut text up*. This lecture is *how to get the text in the first place* — because the cleanest chunker in the world is useless if the extractor turned "five years" into "fi ve ycars" — and *how to prove which chunking strategy actually wins*.

> **Garbage in, garbage indexed.** Extraction is the most-skipped, most-underestimated stage of RAG. A retrieval failure you blame on the embedding is, half the time, an extraction failure two stages upstream. The mantra from Lecture 1 holds: walk *up* the pipeline.

## Part 1 — The extraction toolchain: choosing by document type

There is no universal document extractor, and reaching for the heaviest tool on every document is as wrong as reaching for the lightest. You choose by what the document *is*. Four tools cover almost everything you'll meet in 2026.

### 1.1 PyMuPDF — the fast path for native-text PDFs

A **native-text PDF** is one where the text is real text (selectable, copy-pasteable), not a picture of text. Most PDFs generated by software — Word exports, LaTeX output, contracts from a document system — are native-text. For these, **PyMuPDF (`pymupdf`, imported as `fitz` or `pymupdf`)** is the right tool: it's C-backed, extremely fast, and extracts the embedded text directly with no model inference at all.

```python
import pymupdf  # the modern import name; `import fitz` still works

doc = pymupdf.open("services_agreement.pdf")
full_text = []
for page in doc:
    full_text.append(page.get_text("text"))   # plain reading-order text
document = "\n".join(full_text)
```

`get_text()` has modes that matter:

- `"text"` — plain text in reading order. Your default.
- `"blocks"` — text grouped into layout blocks `(x0, y0, x1, y1, text, ...)`. Use this when you need position (to drop headers/footers by their y-coordinate, or to detect columns).
- `"dict"` — the full structure: blocks → lines → spans, with font, size, and bbox. Use this to detect headings (bigger/bolder font) for the metadata stage.

PyMuPDF also finds tables natively (`page.find_tables()`), which we'll use in §3. It is the first tool you try on any PDF, and for a large fraction of corpora it's the *only* tool you need. The test: can you select text in the PDF viewer? If yes, PyMuPDF will extract it in milliseconds per page.

What it can't do: it can't read a **scanned** PDF (a photo of a page has no embedded text — `get_text()` returns empty), and it doesn't reconstruct reading order on genuinely complex multi-column scientific layouts as well as the specialized tools. For those, you go heavier.

### 1.2 Unstructured — messy mixed documents, typed elements

**Unstructured (`unstructured`)** is the tool for the *messy middle*: documents that mix prose, headings, lists, tables, and footers, and corpora that mix *formats* (PDF + HTML + email + PowerPoint + Markdown). Its distinguishing feature is that it doesn't return a wall of text — it returns **typed elements**:

```python
from unstructured.partition.pdf import partition_pdf

elements = partition_pdf("messy_report.pdf")
for el in elements:
    print(type(el).__name__, "|", str(el)[:60])
# Title          | Services Agreement
# NarrativeText  | This Agreement is entered into between...
# ListItem       | 9. All confidential information must be protected...
# Table          | <html table>
# Footer         | Page 4 of 30
```

Each element carries a category (`Title`, `NarrativeText`, `ListItem`, `Table`, `Header`, `Footer`, `Image`) and metadata (page number, coordinates, parent section). This typing is gold for the pipeline:

- **Cleaning becomes trivial** — drop every `Header`/`Footer`/`PageBreak` element and the repeating boilerplate is gone, no regex.
- **Chunking becomes structure-aware for free** — Unstructured ships `chunk_by_title`, which groups elements under their section heading into chunks, so a chunk is "a section," and the heading is attached as metadata. That's the metadata-injection stretch goal, built in.
- **Tables are isolated**, not smeared into the prose, so you can handle them specially (§3).

```python
from unstructured.chunking.title import chunk_by_title

chunks = chunk_by_title(elements, max_characters=2000, combine_text_under_n_chars=200)
```

The cost: it's much slower than PyMuPDF (it runs layout models for the hard partitioning strategies) and heavier to install. Use it when the document's *structure* is the value — when you want headings, lists, and tables typed and separated — not when you just need the text out of a clean PDF.

### 1.3 MinerU — scientific and complex layouts

**MinerU** (OpenDataLab) is the specialist for **scientific papers and dense technical PDFs**: multi-column layouts, mathematical formulas, figures with captions, and the reading-order problem (in a two-column paper, naive extraction interleaves the columns into nonsense). MinerU runs layout-detection and reading-order models to reconstruct the document into clean Markdown with formulas as LaTeX and tables preserved.

```bash
# MinerU ships a CLI; it's the standard way to run it.
pip install -U "mineru[core]"
mineru -p paper.pdf -o ./out          # -> out/paper.md  (+ extracted figures, JSON layout)
```

Reach for MinerU when: the document is a research paper, a textbook page, a multi-column technical report, or anything where **reading order** and **formulas** matter and PyMuPDF's plain extraction comes out scrambled. It's heavier than Unstructured and benefits from a GPU, but on the documents it's built for, nothing in the lighter tier comes close.

### 1.4 LlamaParse — LLM-powered parsing for the hard documents

**LlamaParse** (LlamaIndex) is the tool you escalate to when the document is genuinely *hard*: financial filings with nested tables, scanned forms, invoices, documents where the *layout itself carries meaning* (a balance sheet's column alignment, a form's field-label-value structure). It uses an LLM to parse the page, which lets it handle structure the rule-based tools choke on — at the cost of latency, an API key, and per-page billing (it has a free monthly quota).

```python
from llama_parse import LlamaParse

parser = LlamaParse(result_type="markdown")   # needs LLAMA_CLOUD_API_KEY
documents = parser.load_data("complex_10k_filing.pdf")
markdown = documents[0].text
```

The escalation ladder, in order of cost and capability:

| Tool | Best for | Speed | Cost | Needs |
|---|---|---|---|---|
| **PyMuPDF** | native-text PDFs (most documents) | fastest | free | nothing |
| **Unstructured** | mixed/messy docs, typed elements, many formats | medium | free | layout deps |
| **MinerU** | scientific/multi-column, formulas, reading order | slow | free | GPU helps |
| **LlamaParse** | hard layouts, nested tables, scanned forms | slowest | free tier then paid | API key |

> **The decision rule:** start at the top of the ladder and only escalate when the cheaper tool *demonstrably* fails on *your* documents (scrambled reading order, missing tables, garbled formulas). Don't pay LlamaParse prices for a contract PyMuPDF reads perfectly. And always *look at the extracted text* before you chunk it — extraction failures are visible to the eye and invisible to the metrics until retrieval is already broken.

## Part 2 — OCR: when there's no text to extract

PyMuPDF returns empty on a scanned PDF because there *is* no embedded text — the page is an image of text. You need **OCR (Optical Character Recognition)** to turn pixels into characters. Two tools matter in 2026.

### 2.1 Tesseract — the CPU workhorse

**Tesseract** (via `pytesseract`) is the decades-old, pure-CPU, no-GPU-required OCR engine. It's the right default for straightforward scanned pages: clean, single-column, reasonable resolution.

```python
import pymupdf
import pytesseract
from PIL import Image
import io

doc = pymupdf.open("scanned_contract.pdf")
page = doc[0]
# Render the page to an image at 300 DPI — OCR accuracy depends on resolution.
pix = page.get_pixmap(dpi=300)
img = Image.open(io.BytesIO(pix.tobytes("png")))
text = pytesseract.image_to_string(img, lang="eng")
```

Two things make or break Tesseract output:

- **Resolution.** OCR at 150 DPI is mush; 300 DPI is the standard floor. Render the page high-res before you OCR it.
- **Preprocessing.** Deskew, binarize (threshold to black/white), and remove speckle, and accuracy jumps. A crooked, gray scan is where Tesseract fails; a clean, straight, high-contrast one is where it shines.

Tesseract's weakness is **layout**: it reads top-to-bottom, left-to-right, so on a multi-column page or a page with a table it interleaves text into the wrong order. For simple pages it's perfect and free; for complex ones it's the wrong tool.

### 2.2 Surya — modern, layout-aware OCR

**Surya** (Datalab) is the modern alternative: a deep-learning OCR + layout-analysis toolkit that does **line detection, layout detection, reading-order detection, and table recognition** — exactly the things Tesseract is blind to. On a multi-column scan, Surya figures out the columns and the reading order *first*, then OCRs in the right sequence. It prefers a GPU but has a CPU fallback (slower).

```python
from PIL import Image
from surya.recognition import RecognitionPredictor
from surya.detection import DetectionPredictor

image = Image.open("scanned_paper_page.png")
det = DetectionPredictor()
rec = RecognitionPredictor()
predictions = rec([image], det_predictor=det)   # layout-aware OCR
for line in predictions[0].text_lines:
    print(line.text)
```

The choice, plainly:

- **Tesseract** — clean single-column scans, no GPU, you want it free and fast and good-enough. The workhorse.
- **Surya** — multi-column, tables, complex layout, rotated text, where reading order matters and you can spare a GPU. The modern, layout-aware option.

Run *both* on a hard scan and diff the output (that's a README stretch goal and Exercise 1's third path): you'll see exactly where Tesseract's reading-order blindness costs you, and decide per-corpus.

## Part 3 — Tables and multimodal pages

### 3.1 Tables

A table is the worst thing you can hand a naive chunker. Flatten a table to plain text and the rows and columns smear together — "Fee $50,000 Insurance $1,000,000 Term 12 months" — and the *structure that made it meaningful* is gone. The relationship "the **insurance** amount is **$1,000,000**" is destroyed when the header and the value end up in different chunks or run together.

The fix is to extract tables **as tables** and chunk them as units:

```python
import pymupdf

doc = pymupdf.open("services_agreement.pdf")
page = doc[3]
tables = page.find_tables()             # PyMuPDF's built-in table finder
for tbl in tables.tables:
    rows = tbl.extract()                # list of rows, each a list of cells
    # Serialize to Markdown so the embedding sees header-to-value structure.
    header, *body = rows
    md = "| " + " | ".join(str(c) for c in header) + " |\n"
    md += "| " + " | ".join("---" for _ in header) + " |\n"
    for r in body:
        md += "| " + " | ".join(str(c or "") for c in r) + " |\n"
    # `md` is now one self-contained chunk: a table that kept its structure.
```

The principles:

- **A table is its own chunk** (or one chunk per row for a long table). Don't let table text bleed into surrounding prose chunks.
- **Serialize to a structure-preserving format** — Markdown or HTML — so the embedding sees the header→value relationship, not a word salad. Unstructured and LlamaParse already return tables as HTML; PyMuPDF/`pdfplumber`/`camelot` give you the cells and you serialize them.
- **Keep the table's caption/heading as metadata** so "Schedule A — Fees" is attached to the fee table even if the cells don't repeat it.

### 3.2 Multimodal pages — text + figure + caption

A page that's text plus a figure plus a caption is three different things, and they want different treatment. The 2026 default is *not* to throw the figure away:

- **Caption with the figure.** A figure's caption is the most retrievable text about it — keep them together, and attach the caption as the figure chunk's text.
- **Optionally, describe the figure with a vision model.** A multimodal LLM (Claude, GPT-4o-class) can generate a text description of a chart or diagram; that description becomes a searchable chunk pointing at the image. This is how you make "the figure showing revenue growth" retrievable when the figure itself isn't text.
- **Keep the image reference in metadata** so retrieval can surface the actual figure to the generator (multimodal RAG), not just its description.

You won't build full multimodal RAG this week — that's later — but the principle starts now: **don't silently drop the non-text content.** A pipeline that extracts only the prose and discards every table and figure has thrown away exactly the content that's often the answer. The extraction stage's job is to *account for everything on the page* and route each part to the right handler.

## Part 4 — Cleaning: the stage between extraction and chunking

Before you chunk, clean. Extraction leaves predictable garbage that, left in, pollutes every chunk:

- **Repeating headers/footers** — "CONFIDENTIAL — Page 4 of 30" on every page. Embedded into every chunk, it adds noise to every vector and can even cause spurious matches (a query about "page" or "confidential"). Drop it (Unstructured's `Footer`/`Header` types, or PyMuPDF `"blocks"` by y-coordinate).
- **Hyphenation across line breaks** — "termi-\nnation" must rejoin to "termination" or the token "termination" (the one your query matches) literally doesn't exist in the chunk.
- **De-column'd interleaving** — fixed by the right extractor (MinerU/Surya), but verify.
- **Whitespace and control characters** — collapse runs of spaces/newlines; strip form-feed and other control chars.

```python
import re

def clean(text: str) -> str:
    text = re.sub(r"(\w)-\n(\w)", r"\1\2", text)     # rejoin hyphenated breaks
    text = re.sub(r"\n{3,}", "\n\n", text)            # collapse blank runs
    text = re.sub(r"[ \t]{2,}", " ", text)            # collapse spaces
    return text.strip()
```

Cleaning is unglamorous and it's where a surprising amount of retrieval quality is won or lost. A clause that reads "fi ve years" after a bad OCR pass will never match a query about "five years," no matter how good your chunker and embedder are.

## Part 5 — The chunking A/B harness

Now the methodology that makes "which chunker?" a measurement instead of an argument. This is the syllabus deliverable and the mini-project's spine.

### 5.1 The one-variable-at-a-time discipline

The entire validity of the A/B rests on **changing exactly one thing**. Fix everything you measured last week:

- **One embedding** — BGE-large-en-v1.5, normalized. (Late chunking is the one exception — it *requires* a long-context model — and you note that asymmetry honestly when you compare it.)
- **One store** — pgvector, `vector_cosine_ops`, `<=>`, the same `ef_search`.
- **One gold set** — the 40-query legal gold set from week 7, unchanged.
- **One metric suite** — Recall@5 and MRR (the spine) plus faithfulness (the secondary signal).

The *only* thing that varies is the chunker. If you change the chunker *and* the embedding, you've learned nothing — you can't attribute the delta. This is the same discipline as week 7's embedding bakeoff, pointed at a different variable.

### 5.2 The harness reuses week 7's `evaluate()` unchanged

This is the payoff of keeping `evaluate()` pure last week. `evaluate(gold, retrieve_fn, k=5)` takes a gold set and a *retrieve function* and returns `{"queries", "top1", "Recall@k", "MRR"}` — it knows nothing about chunking. So the chunking A/B is: for each strategy, build a *different `retrieve_fn`* (one that chunks the corpus that strategy's way, embeds the chunks, indexes them, and retrieves), then call the *same* `evaluate()`:

```python
from crunchrag_embed.eval import evaluate          # week 7, unchanged
from crunchrag_embed import store                  # week 7, unchanged

def make_retriever(chunker, embedder, table_name):
    """Build a retrieve_fn for ONE chunking strategy. Everything else is fixed."""
    chunks = []                                     # (chunk_id, source_clause_id, text)
    for clause_id, text in corpus:
        for i, ch in enumerate(chunker(text)):
            chunks.append((f"{clause_id}::{i}", clause_id, ch))

    store.create_table(table_name, dim=1024)
    vecs = embedder.embed_documents([c[2] for c in chunks])
    store.insert(table_name, [(c[0], c[2], v) for c, v in zip(chunks, vecs)])
    store.build_hnsw(table_name)

    chunk_to_clause = {c[0]: c[1] for c in chunks}

    def retrieve_fn(query: str) -> list[str]:
        qv = embedder.embed_query(query)
        hit_chunks = store.knn(table_name, qv, k=20)
        # Map chunk hits back to clause ids (the gold set is in clause ids),
        # de-duplicated, preserving rank order.
        seen, clause_ranking = set(), []
        for cid in hit_chunks:
            clause = chunk_to_clause[cid]
            if clause not in seen:
                seen.add(clause)
                clause_ranking.append(clause)
        return clause_ranking

    return retrieve_fn

# The A/B loop: same evaluate(), one retrieve_fn per strategy.
for name, chunker in STRATEGIES.items():
    retrieve_fn = make_retriever(chunker, bge, f"chunks_{name}")
    metrics = evaluate(gold, retrieve_fn, k=5)
    print(f"{name:12s}  Recall@5={metrics['Recall@k']:.3f}  MRR={metrics['MRR']:.3f}")
```

The load-bearing trick is the **chunk-to-source mapping**. Your gold set is in *clause ids* (clause_14, clause_09), but you now retrieve *chunks*. So each chunk records which clause it came from, and the retriever maps chunk hits back to clause ids before scoring. A strategy "finds clause 9" if any of clause 9's chunks ranks in the top-k. This is exactly how a real RAG system works — you retrieve chunks but you care about whether the *answer's source* was found.

### 5.3 Faithfulness — the secondary signal, kept honest

Recall@5 and MRR measure *retrieval*: did the right chunk come back? **Faithfulness** measures something retrieval can't: given the retrieved chunks, is the *generated answer* actually grounded in them, or did the model make it up? A strategy can have great Recall@5 but if its chunks are split mid-clause, the generator gets half-answers and hallucinates the rest — faithfulness catches that.

The honest, lightweight way to measure it (LLM-as-judge / Ragas-style):

1. For each gold query, retrieve the top-k chunks with the strategy under test.
2. Generate an answer from those chunks (your generator from earlier weeks).
3. Ask a judge LLM: "Is every claim in this answer supported by these chunks? Score 0–1." (Ragas's `faithfulness` decomposes the answer into claims and checks each against the context.)
4. Average across the gold set → the strategy's faithfulness.

```python
# Lightweight faithfulness sketch (LLM-as-judge). The retrieval metrics are the
# spine; this is the tie-breaker and the "is the answer survivable?" check.
def faithfulness(answer: str, context_chunks: list[str], judge) -> float:
    prompt = (
        "Context:\n" + "\n---\n".join(context_chunks) +
        f"\n\nAnswer:\n{answer}\n\n"
        "Score 0.0-1.0: is EVERY claim in the answer supported by the context? "
        "Reply with only the number."
    )
    return float(judge.complete(prompt).strip())
```

Keep faithfulness in its place: it's the **secondary** metric and the **tie-breaker**. If two strategies have similar Recall@5/MRR, the one with higher faithfulness is the better chunker — its chunks hold *complete*, generatable answers, not just retrievable fragments. But faithfulness is noisier and more expensive (it needs generation + judging), so the retrieval metrics lead and faithfulness confirms. Never let a faithfulness number override a clear Recall@5 loss; a strategy that doesn't retrieve the answer can't be faithful to it.

### 5.4 Picking a winner with reasons

The output of the A/B is not a number — it's a **decision with a justification**. The memo (the homework headline) states: the winning strategy, its Recall@5/MRR/faithfulness, *why* it won on *this* corpus (e.g. "recursive at 512/64 keeps each clause intact because clauses are paragraphs; fixed-512 splits ~15% of them; late chunking matched recursive's recall but cost a long-context model for no gain on these self-contained clauses"), and the trade-off you accepted. A winner you can't explain is a winner you got lucky with.

That's the "the answer survived the chunking" promise, made measurable:

```
$ python chunk_ab.py --strategy recursive --size 512 --overlap 64
strategy=recursive size=512 overlap=64
  Recall@5: 0.88  MRR: 0.74  faithfulness: 0.81
  q12 ("five-year confidentiality duration") -> chunk_09 (rank 1) ✓
     "...confidential information must be protected for five years after termination."
```

The clause survived the cut, in one chunk, and the metric proves it.

## Part 6 — The extraction-to-retrieval decision tree

When retrieval is bad next week, walk this tree — it covers extraction *and* cleaning *and* chunking:

```
Recall@5 is bad on a query whose answer you KNOW is in the corpus.
│
├─ Open the RAW extracted text for the source document. Is the answer there, readable?
│   ├─ No  → EXTRACTION failure. Wrong tool (scanned PDF through PyMuPDF?),
│   │        garbled OCR ("fi ve"), scrambled reading order. Fix the extractor. (§1-2)
│   └─ Yes ↓
│
├─ Is the answer mangled by boilerplate / hyphenation / smeared tables?
│   ├─ Yes → CLEANING failure. Rejoin hyphens, drop headers, isolate tables. (§3-4)
│   └─ No  ↓
│
├─ Print the CHUNKS for that document. Is the whole answer inside ONE chunk?
│   ├─ No  → CHUNKING failure. Answer split across a boundary, or buried in a giant
│   │        chunk. Change strategy/size/overlap and re-measure. (Lecture 1 §7)
│   └─ Yes ↓
│
├─ Does the chunk carry the right METADATA (heading, clause id) for filtering/boosting?
│   ├─ No  → metadata stage. Inject the heading; re-measure. (Lecture 1 §8)
│   └─ Yes ↓
│
└─ Answer is in one well-formed chunk but still not retrieved →
       NOW suspect embedding/index (week 7 territory: prefix, normalization, ef_search).
```

Read the order: you suspect the embedding **last**, not first. That's the corollary mantra as a procedure — *a retrieval failure is more often a chunking (or extraction) failure than an embedding failure* — so you walk up the pipeline from the source text, and only blame BGE when everything upstream checks out.

## Part 7 — Recap

You should now be able to:

- **Choose** the extraction tool by document type: PyMuPDF for native-text PDFs (fast, free, your default), Unstructured for messy mixed docs with typed elements, MinerU for scientific/multi-column/formula-heavy layouts, LlamaParse for hard nested-table/scanned-form documents — and escalate only when the cheaper tool demonstrably fails.
- **Run OCR** with Tesseract (CPU workhorse for clean single-column scans) and Surya (layout-aware, reading-order-aware, GPU-preferred for multi-column/tables), and know to render at 300 DPI and preprocess.
- **Handle tables** as their own structure-preserving chunks (Markdown/HTML, header→value intact) and **multimodal pages** by keeping captions with figures and not silently dropping non-text content.
- **Clean** between extraction and chunking — rejoin hyphenation, drop repeating headers/footers, collapse whitespace — because garbage upstream is garbage indexed.
- **Run the chunking A/B harness**: fix embedding + store + gold set, vary only the chunker, reuse week 7's pure `evaluate()` with a per-strategy `retrieve_fn` and a chunk-to-clause mapping, report Recall@5/MRR (the spine) plus faithfulness (the tie-breaker), and **pick a winner with reasons**.
- **Walk the extraction → cleaning → chunking → metadata → embedding decision tree**, suspecting the embedding last.
